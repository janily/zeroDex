# DEX 业务入门与 zeroDex 业务理解

## 1. 写给传统前端开发者

如果你是从传统前端转入 DApp 前端，最大的变化不是技术栈从 REST API 变成合约 ABI，而是业务系统的信任模型变了。

在传统 Web 应用里，前端通常围绕后端服务工作：

- 用户登录后拿到 token。
- 前端调用后端接口。
- 后端读写数据库。
- 后端决定业务是否成功。
- 前端展示接口返回结果。

在 DApp 里，很多核心业务状态不在项目方数据库中，而在区块链上：

- 用户身份通常是钱包地址。
- 资产余额由 Token 合约记录。
- 交易、流动性、头寸由 DEX 合约记录。
- 写操作需要用户用钱包签名。
- 状态变更要等链上交易被打包确认。
- 一旦交易成功，结果很难像传统数据库一样随便回滚。

所以 DApp 前端不只是“把 API 换成合约调用”。它更像是在帮助用户理解和操作一套公开、异步、不可逆、强金融属性的业务系统。

## 2. DEX 解决什么问题

DEX 是 Decentralized Exchange，去中心化交易所。它解决的是：用户如何在不依赖中心化平台托管资产的情况下，完成 Token 之间的交换。

在中心化交易所中，用户通常先把资产充值到交易所，交易所内部维护账户余额和订单簿。用户下单后，交易所的撮合系统撮合买卖双方，成交结果记录在交易所数据库里。用户真正链上提币时，才发生链上转账。

DEX 的目标不同：

- 用户资产仍在自己的钱包里。
- 交易通过智能合约执行。
- 成交规则公开写在合约里。
- 任何人都可以验证交易结果。
- 项目方不能随意改用户余额。

这也是为什么 DEX 前端里会频繁出现钱包连接、授权、交易签名、等待确认这些概念。前端不是在替用户操作账户，而是在构造一笔链上交易，让用户确认后提交到区块链。

## 3. 从订单簿到 AMM

传统交易所常见模型是订单簿。比如有人挂单：

- 愿意用 100 USDT 买 1 ETH。
- 愿意用 101 USDT 卖 1 ETH。

撮合系统会根据价格和时间优先级完成成交。

但是链上订单簿有几个问题：

- 每次挂单、撤单、成交都可能需要链上交易，成本高。
- 高频撮合不适合大多数公链性能。
- 用户体验会受 Gas 和确认时间影响。

AMM，也就是 Automated Market Maker，自动做市商，是 DEX 中更常见的方案。

AMM 不依赖买卖双方同时出现，而是让用户和“流动性池”交易。池子里提前放好了两种 Token，例如 MNTokenA 和 MNTokenB。用户想用 A 换 B，就把 A 交给池子，从池子里取走 B。

价格不是由订单簿撮合出来的，而是由池子内的资产比例和算法计算出来的。

## 4. AMM 的核心业务对象

### 4.1 Token

Token 是链上的资产合约。对前端来说，最常用的是 ERC20。

你需要关心：

- `symbol`：展示名，例如 MNTokenA。
- `decimals`：精度，例如 18 位。
- `balanceOf`：用户余额。
- `allowance`：用户授权给某个合约可花费的额度。
- `approve`：用户授权某个合约使用自己的 Token。

传统前端里，余额可能来自后端接口。DApp 里，余额来自 Token 合约。

### 4.2 交易对

交易对表示两个 Token 之间可以交换，例如：

- MNTokenA / MNTokenB
- MNTokenA / MNTokenC

用户在 Swap 页面选择“支付 Token”和“获得 Token”，本质上是在选择一个交易对和交易方向。

### 4.3 Pool

Pool 是流动性池。它保存两种 Token 的资金，并根据算法提供兑换。

在 zeroDex 里，一个交易对可以有多个 Pool。它们可能有不同：

- 费率。
- 价格范围。
- 当前价格。
- 流动性规模。

这意味着用户选择 MNTokenA 换 MNTokenB 时，不一定只有一个池子可以交易。前端需要比较多个池子的报价，帮用户找到更好的路径。

### 4.4 Liquidity

Liquidity 是流动性。它可以理解为池子中可用于成交的资金能力。

流动性越高，通常用户大额交易时价格变化越小。流动性越低，用户稍微换多一点，价格就可能明显变差，甚至无法完全成交。

传统前端做电商时会关心库存。DEX 中，流动性有点像“可成交库存”，但它不是固定价格库存，而是会随着交易改变价格。

### 4.5 LP

LP 是 Liquidity Provider，流动性提供者。

LP 把两种 Token 放进池子里，让别人可以交易。作为回报，LP 可以获得交易手续费。

从业务上看，LP 不是简单存款，而是在做市：

- 如果别人频繁使用这个池子交易，LP 可以获得手续费。
- 如果价格变化很大，LP 的资产组合可能发生变化。
- LP 承担价格波动风险。

### 4.6 Position

Position 是 LP 的头寸，表示某个用户在某个池子里拥有多少流动性，以及可以领取多少手续费或代币。

在 MetaNodeSwap 设计中，`PositionManager` 使用 ERC721 NFT 管理 LP 头寸。也就是说，一个 LP 头寸可以被表示成一个 NFT。

对前端来说：

- 用户持有哪些 positionId，可以通过 NFT 查询服务获取。
- 每个 positionId 可以查到对应交易对、池子 index、流动性、可领取金额。
- burn 是移除流动性。
- collect 是领取可提取的 Token 和手续费。

## 5. 价格、滑点与价格影响

### 5.1 价格

在普通交易产品里，用户看到的是“1 A = 多少 B”。这是业务价格。

但在合约中，价格可能用 tick、sqrtPriceX96 等形式表示。这些表示适合链上计算，不适合直接展示给用户。

zeroDex 前端应该把底层价格转换成用户能理解的汇率：

- 当前价：`1 MNTokenA = 2.5 MNTokenB`
- 反向价：`1 MNTokenB = 0.4 MNTokenA`
- 价格范围：`1 MNTokenA = 2.0 - 3.0 MNTokenB`

### 5.2 滑点

滑点是用户愿意接受的成交结果偏差。

例如报价显示：

- 输入 100 A，预计获得 250 B。

如果用户设置 1% 滑点，那么最低可接受输出是：

- 247.5 B。

如果链上实际成交时只能获得 246 B，这笔交易应该失败，避免用户在更差价格下成交。

滑点存在的原因是：报价发生在交易提交前，而链上真正执行发生在交易被打包时。中间池子状态可能被别人交易改变。

### 5.3 价格影响

价格影响是用户自己的交易对池子价格造成的影响。

如果池子很小，用户用大量 A 换 B，会让池子里的 A 变多、B 变少，A 相对 B 的价格变差。用户得到的平均成交价会低于当前显示价格。

前端需要让用户知道：报价不是一个固定柜台价，而是基于池子状态和交易规模计算出的结果。

## 6. Approve 不是支付

这是 DApp 前端最容易误解的地方之一。

用户调用 `approve` 时，并没有把 Token 转给 DEX。Approve 的业务含义是：

“我允许某个合约最多从我的钱包里使用一定数量的某个 Token。”

真正的 Token 转移发生在后续的 `swap` 或 `mint` 中，由合约通过 `transferFrom` 拉取用户 Token。

所以前端上经常会出现两步：

1. Approve：授权合约使用 Token。
2. Swap 或 Mint：执行真实业务操作。

在 zeroDex 中：

- Swap 前，用户要把 `tokenIn` approve 给 `SwapRouter`。
- Mint 前，用户要把 `token0` 和 `token1` approve 给 `PositionManager`。

授权给谁非常重要。授权目标错了，后续交易会失败；授权给不可信合约，也可能带来资产风险。

## 7. MetaNodeSwap 的业务模型

MetaNodeSwap 是一个教学和简化版 DEX 合约设计。它借鉴了 Uniswap V3 的一部分思想，但做了明显简化。

### 7.1 固定价格范围的池子

MetaNodeSwap 中，每个池子都有自己的价格范围。

例如某个 MNTokenA/MNTokenB 池子只允许在：

`1 A = 2 B` 到 `1 A = 3 B`

这个范围内成交。

当交易把价格推到边界时，这个池子就不能继续提供更远价格的成交。如果用户交易还没有完全满足，前端可以继续尝试同交易对下的其他池子。

这和 Uniswap V3 的完整设计不同。Uniswap V3 中，每个 LP 可以选择自己的价格区间；而 MetaNodeSwap 简化为池子有固定价格区间，所有向该池子添加流动性的 LP 都使用同一个区间。

### 7.2 同一交易对可以有多个池子

MetaNodeSwap 允许相同交易对和费率存在多个池子，只要它们的价格范围等参数不同。

业务含义是：市场被拆成多个价格区间的流动性场所。

用户 Swap 时，前端不应该只看一个池子，而应该：

- 找到同交易对的全部池子。
- 排除无流动性池子。
- 对每个可用池子报价。
- 选择更优成交路径。

### 7.3 创建池子

任何人都可以创建池子。创建池子时要指定：

- token0。
- token1。
- 费率。
- 价格范围。
- 初始价格。

创建池子本身只是创建一个市场场所。池子如果没有流动性，用户仍然无法有效交易。

所以业务上通常是：

1. 创建池子。
2. 添加流动性。
3. 其他用户开始 Swap。

### 7.4 添加流动性

LP 选择一个已有池子，存入两种 Token。

在 zeroDex 中，添加流动性时用户不能填写 tick low 或 tick high，因为价格范围已经由池子决定。

这点非常重要：Position 页面不是让用户创建任意区间头寸，而是让用户加入某个已有池子的统一价格范围。

### 7.5 Swap

Swap 是用户用一种 Token 换另一种 Token。

MetaNodeSwap 支持两种模式：

- Exact Input：用户确定支付多少，系统计算最多能获得多少。
- Exact Output：用户确定想获得多少，系统计算最少需要支付多少。

Exact Input 更像“我有 100 A，帮我换成尽可能多的 B”。

Exact Output 更像“我需要 250 B，帮我计算最多要花多少 A”。

### 7.6 手续费

用户 Swap 时会支付手续费。手续费按 LP 提供的流动性份额分配。

业务上可以理解为：

- LP 给池子提供资金。
- 交易用户使用池子完成兑换。
- 交易用户支付手续费。
- LP 按贡献比例获得手续费收入。

在前端中，手续费收益通常体现在 Position 的可领取金额里。用户通过 collect 领取。

## 8. zeroDex 三个页面的业务含义

### 8.1 Swap 页面

Swap 页面是交易用户的入口。

用户关心的是：

- 我支付什么 Token。
- 我获得什么 Token。
- 我能获得多少。
- 价格是否合理。
- 是否需要授权。
- 交易是否成功。

业务本质是：

- 前端寻找合适池子。
- 前端用静态调用获取报价。
- 用户授权 Token。
- 用户签名提交 swap 交易。
- 合约从用户钱包拉取输入 Token，并从池子转出输出 Token。

前端需要特别注意：报价不等于最终成交。报价是基于当前链上状态模拟出来的结果，真实成交要等交易执行。

### 8.2 Pool 页面

Pool 页面是市场列表。

用户看到的是有哪些交易场所可以使用：

- 哪些 Token 可以交易。
- 当前价格是多少。
- 价格范围是多少。
- 流动性是否充足。
- 费率是多少。

对于 LP 来说，Pool 页面也是选择做市机会的地方。一个池子如果价格范围合理、交易活跃，可能带来手续费收入。

创建池子是更高级的操作。创建者相当于定义了一个新的交易价格区间，但只有添加流动性后，这个池子才真正有用。

### 8.3 Position 页面

Position 页面是 LP 的资产管理页。

用户关心的是：

- 我在哪些池子提供了流动性。
- 每个头寸对应多少流动性。
- 我能领取多少 Token 或手续费。
- 我能否退出这个头寸。

业务本质是：

- Position NFT 代表 LP 在某个池子的权益。
- burn 移除流动性，把可提取资产记入头寸。
- collect 把可提取资产转回用户钱包。

这和传统理财产品的“持仓页”有点像，但区别是这里的持仓状态来自链上合约，操作也需要钱包签名。

## 9. 一次完整 Swap 的业务链路

下面用 Exact Input 举例：用户用 MNTokenA 换 MNTokenB。

### 9.1 连接钱包

用户连接 MetaMask。前端拿到用户地址。

业务意义：确认“谁在操作”。

### 9.2 检查网络

前端检查用户是否在 Sepolia。

业务意义：zeroDex 合约部署在 Sepolia，如果用户在其他网络，读写的是另一套链上世界。

### 9.3 选择 Token

用户选择支付 MNTokenA，获得 MNTokenB。

业务意义：确定交易对和交易方向。

### 9.4 输入数量

用户输入 100 MNTokenA。

业务意义：用户选择 Exact Input，目标是在给定输入下最大化输出。

### 9.5 获取池子

前端调用 PoolManager 获取所有池子，筛选 MNTokenA/MNTokenB 相关池子。

业务意义：找到可用交易场所。

### 9.6 静态报价

前端对候选池子或路径调用 SwapRouter 的 quote 方法，使用 staticCall 或 simulateContract。

业务意义：模拟如果现在交易，大约能获得多少 MNTokenB。

这里不能发送真实交易。报价只是试算。

### 9.7 选择最优路径

前端比较报价，选择输出最多的路径。

业务意义：帮助用户获得更好成交结果。

### 9.8 检查余额

前端读取 MNTokenA 余额。

业务意义：确认用户有足够资产支付。

### 9.9 检查授权

前端读取 MNTokenA 对 SwapRouter 的 allowance。

业务意义：确认 SwapRouter 是否有权限从用户钱包拉取 MNTokenA。

### 9.10 Approve

如果授权不足，用户先签名发送 approve 交易。

业务意义：用户授权 SwapRouter 使用 MNTokenA。

### 9.11 Swap

授权足够后，用户签名发送 exactInput 交易。

业务意义：合约实际执行兑换，Token 发生转移。

### 9.12 等待确认

交易进入 pending，等待区块打包。

业务意义：链上状态还没有最终变化，前端不能提前当作成功。

### 9.13 刷新状态

交易确认后，前端刷新余额、池子状态和报价。

业务意义：展示新的链上真实状态。

## 10. 一次完整添加流动性的业务链路

用户想成为 LP，向某个 MNTokenA/MNTokenB 池子添加流动性。

### 10.1 选择交易对

用户选择 MNTokenA/MNTokenB。

业务意义：选择要在哪个市场做市。

### 10.2 选择池子

前端展示该交易对下已有池子。用户选择一个池子。

业务意义：选择价格范围、费率和当前市场状态。

注意：用户不能自己输入 tickLow 或 tickHigh。价格范围由池子决定。

### 10.3 输入两种 Token 数量

用户输入愿意投入的 MNTokenA 和 MNTokenB 数量。

业务意义：用户准备向池子提供资金。

### 10.4 检查余额和授权

前端分别检查：

- MNTokenA 余额和对 PositionManager 的授权。
- MNTokenB 余额和对 PositionManager 的授权。

业务意义：PositionManager 需要能够从用户钱包拉取两种 Token。

### 10.5 Approve

如果任一 Token 授权不足，用户需要先 approve。

业务意义：授权 PositionManager 使用对应 Token。

### 10.6 Mint

用户调用 PositionManager 的 mint。

业务意义：合约把用户资金加入池子，并铸造一个 LP Position NFT 给用户。

### 10.7 查看头寸

前端通过 ZAN 查询用户持有的 Position NFT，再调用合约查询头寸详情。

业务意义：用户看到自己的 LP 权益。

## 11. Burn 和 Collect 的业务区别

Burn 和 Collect 很容易混淆。

Burn 是移除流动性。它的重点是：

- 不再用这部分资金继续做市。
- 合约计算用户可以取回多少 token0 和 token1。
- 可取回资产会记录到头寸里。

Collect 是领取资产。它的重点是：

- 把头寸中可领取的 token0 和 token1 转回用户钱包。
- 领取内容可能包含移除流动性后的本金，也可能包含手续费。

简单理解：

- burn 是“退出做市并结算应得资产”。
- collect 是“把已结算资产提回钱包”。

## 12. DApp 前端的业务直觉

### 12.1 读操作和写操作完全不同

读操作通常免费、不需要签名，例如查询池子、余额、头寸。

写操作需要用户签名，并可能消耗 Gas，例如 approve、swap、mint、burn、collect。

### 12.2 报价不是承诺

Quote 是模拟当前状态下的结果。交易真正执行时，池子状态可能已经改变。

所以前端需要滑点保护、deadline 和失败提示。

### 12.3 Approve 不是业务完成

Approve 只授权，不兑换、不添加流动性。

用户完成 approve 后，还需要继续执行 swap 或 mint。

### 12.4 Pending 是正常状态

链上交易不是传统接口请求。用户签名后，交易可能 pending 一段时间。

前端必须设计 pending 状态，不能让用户误以为页面卡死。

### 12.5 失败也可能有成本

如果交易已经上链执行但最终 revert，用户仍可能损失 Gas。

所以前端要尽可能在提交前做余额、授权、报价、滑点、参数校验。

### 12.6 链上状态有延迟

交易刚提交后，马上读链可能还是旧状态。应该等交易 receipt 确认后再刷新。

### 12.7 用户需要被保护

DEX 是金融产品。前端不应该只追求“能调通合约”，还要帮助用户理解：

- 要授权给谁。
- 会花多少。
- 最少能得到多少。
- 交易为什么可能失败。
- 当前池子是否流动性不足。

## 13. zeroDex 中常见术语表

### DEX

去中心化交易所。用户通过智能合约完成 Token 交换，资产不需要托管给中心化平台。

### AMM

自动做市商。用算法和流动性池提供交易价格与成交能力。

### Token

链上资产。zeroDex 中使用 4 个 Sepolia 测试 ERC20。

### Pair

交易对，例如 MNTokenA/MNTokenB。

### Pool

流动性池。保存一对 Token 的资金，并支持用户在该池子中交易。

### Liquidity

流动性。表示池子可支持交易的资金能力。

### LP

流动性提供者。把 Token 放入池子，换取手续费收益。

### Position

LP 头寸。表示某个用户在某个池子中的流动性权益。

### Position NFT

用 ERC721 表示的 LP 头寸凭证。用户持有 NFT，代表拥有对应头寸。

### Swap

兑换。用户用一种 Token 换另一种 Token。

### Exact Input

固定输入数量，尽量获得更多输出 Token。

### Exact Output

固定想获得的输出数量，尽量减少输入 Token。

### Quote

报价。前端通过静态调用模拟交易结果，不改变链上状态。

### Approve

授权。用户允许某个合约使用自己一定数量的 Token。

### Allowance

授权额度。表示某个 spender 还能从用户钱包使用多少 Token。

### Slippage

滑点。用户愿意接受的成交结果偏差。

### Price Impact

价格影响。用户这笔交易本身对池子价格造成的影响。

### Deadline

交易截止时间。超过该时间交易不应继续执行，避免过期报价成交。

### Tick

链上价格刻度。适合合约计算，不适合作为主要用户展示。

### sqrtPriceX96

合约中用于表示价格的高精度格式。前端应转换成汇率展示。

### Burn

移除流动性。把 LP 的流动性从池子中撤出，并结算可领取资产。

### Collect

领取。把头寸中已结算的 Token 或手续费转回用户钱包。

## 14. 学习建议

如果你从传统前端切入 DApp 前端，建议按下面顺序理解 zeroDex：

1. 先理解 ERC20：余额、精度、approve、allowance。
2. 再理解钱包：地址、签名、网络、交易确认。
3. 再理解 Swap：报价、滑点、授权、成交。
4. 再理解 Pool：流动性从哪里来，价格为什么会变。
5. 再理解 Position：LP 为什么有 NFT，burn 和 collect 为什么分开。
6. 最后理解 tick 和 sqrtPriceX96：它们是底层价格表示，不是用户业务语言。

对前端开发来说，最重要的不是一开始就把所有数学细节吃透，而是先建立正确业务直觉：

- 用户的钱在钱包里。
- 合约是业务执行者。
- 前端负责解释、模拟、校验和提交交易。
- 链上确认后，业务状态才真正改变。

当你能把每个按钮背后的资产流转讲清楚，就已经进入 DApp 前端开发的核心区域了。
